## Ledger Design Exploration

Started designing the `transactions` table which will serve as the financial ledger for the system.

Key design conclusions:

1. Transactions belong to **accounts**, not users.
2. Amount values will be **always positive**.
3. Transaction type determines behavior:
   - income
   - expense
   - transfer
4. Transfers should be recorded as **two ledger rows**:
   - transfer_out
   - transfer_in

This approach ensures correct balance calculations for each account.

## Credit Card Lifecycle Consideration

Analyzed how credit card purchases and payments should be represented.

Example scenario:

- Purchase occurs on credit card.
- Bill is generated in the next billing cycle.
- Payment is made from savings account.

Conclusion:

Credit card payments should be modeled as **transfers between accounts**.

## Category Model Exploration

Reviewed possible category and subcategory taxonomy for transactions.

Conclusion:

Transactions should reference **subcategory**, not category.

Proposed structure:

categories  
subcategories  
transactions

Relationships:

categories (1) → (many) subcategories  
subcategories (1) → (many) transactions

---

## Investment Flow Modeling

Evaluated how investment activity should be recorded.

Decision direction:

Investment purchases should be modeled as **transfers** rather than expenses.

Example:

Savings → Investment Account

This approach allows tracking of:

- invested capital
- withdrawals
- realized returns
